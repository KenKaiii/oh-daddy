---
name: setup-railway
description: Deploy oh-daddy to Railway end-to-end.
---

Goal: get oh-daddy fully running on Railway with as little friction for the user
as possible. You (the agent) do everything automatable via the Railway CLI.
The deterministic provisioning lives in `scripts/railway-setup.sh`; your job is
preflight, running it, and pointing the user at the in-app Setup wizard.

This script provisions INFRASTRUCTURE ONLY. Meta/Instagram connect credentials
(Instagram app ID/secret, optional Facebook app ID/secret/config ID, and the
webhook verify token generated by the wizard) are NOT passed via env or the CLI —
the user enters them in the app's Settings page / Setup wizard after first login,
where they're stored encrypted in the database.

## 1. Preflight (do silently; only stop if something is missing)

Run these checks:
- `command -v railway` — if missing, tell the user to install the Railway CLI
  (`brew install railway` or https://docs.railway.com/guides/cli) and STOP.
- `railway whoami` — if it errors, the user must authenticate in a browser,
  which you cannot do. STOP and tell them to run `railway login`, then re-invoke.
- `command -v openssl && command -v node` — both ship on macOS; if absent, STOP.

## 2. Optional input (only if needed)

There are NO credentials to collect — do NOT ask for Meta/Instagram app id,
secret, config id, webhook token, encryption key, dashboard password, database
URL, or domain. All of those are either generated by the script or entered later
in the app's Setup wizard.

The only thing you might need:
- **RAILWAY_WORKSPACE** — required ONLY if the account has more than one
  workspace (the script lists them and tells the user which to pass). With a
  single workspace, omit it.

Inngest is SELF-HOSTED and fully automated — do NOT ask for any Inngest keys.
The script provisions a self-hosted Inngest server (Railway template) and wires
the app's signing/event keys + base URL to it via cross-service references.

Tell them plainly: "I'll generate APP_ENCRYPTION_KEY, ADMIN_PASSWORD, and the
self-hosted Inngest keys automatically and store them in Railway. Your Meta and
Instagram credentials go into the app's Setup wizard after login — not here."

## 3. Run the provisioning script

Run the script directly — no credentials to export:

```bash
bash scripts/railway-setup.sh
# or, if the account has multiple workspaces:
RAILWAY_WORKSPACE="<name or id>" bash scripts/railway-setup.sh
```

The script is idempotent — safe to re-run. It will:
- link/create the Railway project (needs RAILWAY_WORKSPACE if >1 workspace) and
  provision Postgres + the app service,
- generate APP_ENCRYPTION_KEY ONCE (never rotated — rotating orphans tokens),
- generate ADMIN_PASSWORD if absent,
- wire DATABASE_URL, drop INNGEST_DEV,
- provision a SELF-HOSTED Inngest server (Railway template) and wire the app's
  INNGEST_SIGNING_KEY / INNGEST_EVENT_KEY / INNGEST_BASE_URL to it via
  cross-service references (always in lockstep with the server),
- mint a public domain and set NEXT_PUBLIC_APP_URL,
- `railway up`, apply `db/schema.sql` via the DB's public proxy URL (local psql
  or a postgres Docker container), and PUT the app's public `/api/inngest` URL to
  sync functions with a publicly reachable Inngest callback URL.

It does NOT set any Meta/Instagram credentials — those are entered in-app.

Do NOT edit scripts/railway-setup.sh while it is running — bash re-reads the
file mid-execution and will misalign. If the script exits non-zero, read its
last lines, fix the specific blocker (e.g. re-run `railway login`, set
RAILWAY_WORKSPACE, or a transient deploy error), and re-run it.

## 4. Migration (only if upgrading an existing deploy)

A fresh deploy starts empty — SKIP this. Only if this Railway DB already holds
plaintext rows from a pre-encryption deploy, encrypt them in place once. The
DB's internal URL isn't reachable locally, so target the public proxy URL with
the app's encryption key:

```bash
DATABASE_URL="$(railway variables --service Postgres --kv | sed -n 's/^DATABASE_PUBLIC_URL=//p')" \
APP_ENCRYPTION_KEY="$(railway variables --service oh-daddy --kv | sed -n 's/^APP_ENCRYPTION_KEY=//p')" \
node db/encrypt-existing-secrets.mjs
```

## 5. Relay the final steps

The script prints a summary. Surface to the user, verbatim and prominently:
- The **dashboard login password** (only plaintext copy — they must save it).
- The app URL — sign in at `https://<domain>/login`.
- Then point them at the in-app **/setup** wizard, which walks through entering
  Instagram app credentials, optional Facebook app credentials/config ID,
  generating the webhook verify token, and showing the exact redirect + callback
  URLs to register in the Meta App dashboard. There is nothing to add to Railway
  by hand.

Keep your final message tight: app URL, the password to save, and "finish in the
/setup wizard". Do not re-print secrets that live only in Railway (e.g.
APP_ENCRYPTION_KEY).
